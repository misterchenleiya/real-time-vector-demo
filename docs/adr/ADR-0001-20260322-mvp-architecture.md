# ADR-0001-20260322-mvp-architecture

- Status: accepted
- Date: 2026-03-22

## 背景与问题定义

`real-time-vector-demo` 需要验证一条从本地图片 / 视频输入，到稳定矢量路径生成，再到多区域 / 多设备网络输出的完整链路。该项目是技术原型，不是终端用户产品，因此核心问题是：

- 核心处理引擎是否能独立于 GUI 存在
- 普通自然视频是否能通过经典 CV 管线得到可用的稳定轮廓
- 场景路径是否能稳定映射为多设备输出帧
- 本地多实例是否能通过最简 UDP 协议实时联调

## 目标

- 提供图片 / 视频输入与实时或准实时处理
- 生成稳定、简化且可预览的矢量路径
- 支持场景到区域、区域到设备的坐标映射
- 支持通过本地回环网络将设备帧分发给多个 DEMO 实例
- 接收端在没有本地媒体时自动显示网络收到的路径
- 提供最小 GUI 和 CLI 两个入口

## 非目标

- 不做产品级 GUI
- 不做交互式区域编辑器
- 不做时间线编辑、插件系统和媒体资源库
- 不做复杂设备协议适配
- 不在 MVP 中强制引入 ONNX Runtime

## 决策结论

采用“核心引擎优先、Qt Widgets 外壳最小化、OpenCV 经典管线优先”的实现策略：

1. 核心处理链使用 OpenCV 进行缩放、去噪、边缘提取、形态学处理和轮廓提取。
2. 轮廓路径统一转换到归一化场景坐标，再进入路径简化、重采样、平滑和跨帧稳定化。
3. 输出模型先收敛为“设备矢量帧（DeviceFrame）”，由发送端根据配置 fan-out 到多个本地 DEMO 实例。
4. 本地网络层采用最简自定义 UDP 二进制协议，并提供分片 / 重组以规避单包大小限制。
5. GUI 仅承担媒体选择、开始 / 停止、参数预览、路径预览和状态展示，核心逻辑不依赖 GUI。

## 方案概述

### 核心模块

- `core/config`：JSON 配置加载
- `core/model`：场景帧、设备帧、几何和指标模型
- `core/processing`：媒体读取、轮廓提取、路径稳定化、发送引擎
- `core/routing`：场景到区域 / 设备的坐标映射
- `core/network`：设备帧编解码、UDP 发送、接收和重组
- `app/gui`：Qt Widgets 最小界面
- `app/cli`：无 GUI 的发送 / 接收入口

### 数据流

发送端：

`图片/视频 -> OpenCV 处理 -> SceneFrame -> 路由到 DeviceFrame -> UDP fan-out -> 本地/远端 DEMO 显示`

接收端：

`UDP 收包 -> 分片重组 -> DeviceFrame 解码 -> 预览`

### 线程与调度

- GUI 模式下，发送引擎使用独立 `QTimer` 驱动处理节拍。
- 网络接收基于 `QUdpSocket` 异步事件。
- MVP 第一版不引入复杂线程池，优先保证结构清晰和功能可调试。

## 涉及模块 / 数据结构 / 接口 / 配置 / 存储影响

- 场景帧统一为归一化路径集合：`SceneFrame`
- 输出帧统一为面向单设备的路径集合：`DeviceFrame`
- JSON 配置包含：
  - `processing`
  - `network`
  - `zones`
- 不引入数据库或持久化存储

## 协议决策

MVP 网络协议不做真正广播，而采用“发送端按配置对多个本地目标逐个 fan-out”的方式，以降低不同平台对 UDP 广播 / 组播行为的差异风险。

每个 `DeviceFrame` 通过二进制协议编码后，按固定上限分片为多个 UDP 包。接收端按 `(sourceId, deviceId, frameId)` 重组，超时未收齐则丢弃。

## 备选方案及放弃原因

### 方案 A：一开始引入 ONNX / 深度分割模型

放弃原因：

- 依赖更重，跨平台构建复杂度明显上升
- 目前目标是验证引擎架构和数据流，而不是追求第一版模型效果上限
- 经典 CV 管线更容易调试和快速迭代

### 方案 B：使用真正的 UDP 广播或组播

放弃原因：

- 本地回环网络下平台差异更大
- 多实例同时监听的可重复性不如明确 fan-out
- MVP 目标是稳定验证，不是网络特性研究

### 方案 C：先做完整 GUI，再补核心引擎

放弃原因：

- 会让演示层绑架核心设计
- 不利于后续做 CLI、自动化验证和无界面运行

## 兼容性、迁移方案与风险

- 发送端依赖 OpenCV；如果环境缺失 OpenCV，只能运行接收预览相关功能
- 自然视频轮廓稳定性是 MVP 最大风险，需通过参数和样例约束控制
- UDP 为无连接协议，必须接受分片丢失导致的帧级丢弃
- 当前开发环境若不是目标架构，需要后续在 Linux / Windows x64 做补充验证

## 验证与回滚思路

验证：

- 固定样例图片 / 视频下能够持续产生路径
- 两个以上接收实例能同时显示发送端输出
- 不加载媒体时自动进入接收模式
- CLI 可无 GUI 运行

回滚：

- 若自然视频处理链不稳定，可将处理链临时回滚到“边缘 + 最大轮廓”的更保守版本
- 若多区域映射不稳定，可先退回到全场景单区域映射
